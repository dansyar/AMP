======================================================================

You are Codex, acting as a senior software architect and hackathon engineering lead.

Help us design a practical project architecture for a hackathon. The goal is to create something we can build quickly, demo reliably, and extend later if needed.

Project context:
- Hackathon duration: [24/36/48 hours]
- Team size: [number of people]
- Project idea: [describe the idea]
- Target users: [who will use it]
- Core demo scenario: [the main flow judges must see working]
- Required features: [must-have features]
- Nice-to-have features: [optional features]
- Preferred tech stack: [if any]
- Deployment target: [Vercel, Render, Railway, Supabase, Firebase, Docker, local demo, etc.]
- External APIs/services: [OpenAI, maps, payments, auth, sensors, etc.]
- Constraints: [time, budget, privacy, offline support, hardware, limited API access, etc.]

Design a hackathon-ready architecture with:

1. High-level architecture
- Frontend
- Backend/API
- Database/storage
- External services
- Deployment setup
- End-to-end user flow

2. Recommended tech stack
- Choose fast, reliable, demo-friendly tools
- Explain why each choice fits the hackathon
- Avoid unnecessary complexity

3. Repository structure
- Propose a clean folder layout
- Include frontend, backend, shared code, scripts, docs, and config where useful

4. Core modules
- Break the system into clear components
- Define responsibilities and boundaries
- Keep business logic separate from UI where reasonable

5. Data model
- Define the minimum required entities/tables/collections
- Include key fields and relationships
- Keep the schema MVP-focused

6. API design
- List the main endpoints or service functions
- Include example request/response shapes for key flows

7. MVP build plan
- Break implementation into hackathon milestones
- Identify what to build first
- Identify what can be mocked, hardcoded, or skipped
- Assign possible tasks across team members

8. Demo strategy
- Explain how to make the demo reliable
- Include seed data, fallback states, local vs deployed demo options, and failure handling

9. Risks and tradeoffs
- Identify likely technical risks
- Suggest simple mitigations

10. Future extensions
- List improvements that should not block the MVP

Be concrete and pragmatic. Prioritize a working vertical slice over a perfect architecture. If details are missing, make reasonable assumptions and clearly state them before giving the architecture.

4. OPTIMIZED PROMPT - AMP AI FLUENCY ARCHITECTURE
======================================================================

You are Codex, acting as a senior product architect, hackathon strategist, and AI programme designer.

We are joining a hackathon/ideathon for the following challenge:

"AI is reshaping every profession. AMP's next step is to build a community that does not just learn about AI, but leads with it. Design an AI Fluency ideathon that surfaces and empowers AI Champions from within the Muslim professional community to pilot, run, and scale grassroots AI learning programmes."

Our task is to design a project architecture and MVP concept that addresses:

- The biggest AI knowledge gaps facing Muslim professionals today
- A peer-led AI fluency programme that can scale across workplaces and community spaces
- A clear definition of what an AI Champion looks like
- A way to measure real AI fluency and behaviour change, not just attendance
- A practical pilot model that AMP can run, evaluate, and expand

Create a hackathon-ready project architecture for a digital platform, programme system, or hybrid solution that helps AMP identify, train, activate, and measure AI Champions.

Assume:
- Target users include Muslim professionals, workplace mentors, community organisers, AMP programme leads, and potential AI Champions
- The solution should be realistic to prototype during a hackathon
- The MVP must be demo-friendly and show a clear end-to-end flow
- The architecture should support future scaling across companies, mosques, community organisations, and professional networks
- The solution should be culturally aware, inclusive, privacy-conscious, and professionally credible

Produce the following:

1. Problem Framing
- Summarise the core problem in practical terms
- Identify the most likely AI knowledge gaps among Muslim professionals
- Distinguish between awareness gaps, practical skill gaps, confidence gaps, ethical concerns, and workplace adoption barriers

2. Proposed Solution
- Describe the product/programme concept
- Explain how it helps AMP surface, train, and empower AI Champions
- Explain what makes it peer-led and scalable
- Include the main value proposition for professionals, employers, community spaces, and AMP

3. High-Level Architecture
- Describe the frontend experience
- Describe the backend/API layer
- Describe the database or storage model
- Describe any AI components
- Describe analytics and measurement components
- Include an end-to-end flow from user onboarding to champion activation and impact tracking

4. MVP Demo Flow
Design one strong judging demo flow, for example:
- A professional takes an AI fluency self-assessment
- The system identifies their knowledge gaps
- The system recommends a peer-led learning pathway
- A potential AI Champion is identified based on readiness signals
- The Champion receives a starter kit to run a session
- Participants complete a practical challenge
- AMP sees dashboard evidence of fluency growth and behaviour change

5. Core Features
Split features into:
- Must-have MVP features
- Nice-to-have features
- Future scaling features

For each feature, explain:
- What it does
- Who uses it
- Why it matters
- Whether it should be built, mocked, or simulated for the hackathon

6. AI Champion Definition
Define what an AI Champion is in this context.
Include:
- Required traits
- Practical skills
- Community leadership behaviours
- Ethical awareness
- Workplace/community influence
- Readiness indicators
- Example scoring rubric

7. AI Fluency Measurement Model
Design a measurement framework that tracks real fluency and behaviour change, not just attendance.
Include:
- Baseline assessment
- Practical task completion
- Confidence and mindset shift
- Real workplace/community use cases
- Peer teaching activity
- Follow-up behaviour tracking
- Champion activation metrics
- Suggested scoring system

8. Data Model
Propose the minimum useful data entities, such as:
- User
- Organisation
- Community Space
- Assessment
- Knowledge Gap
- Learning Pathway
- AI Champion Profile
- Session
- Practical Challenge
- Behaviour Change Evidence
- Impact Metric

For each entity, include key fields and relationships.

9. API / Service Design
Propose the main API endpoints or service functions.
Include example request and response shapes for:
- Submitting an AI fluency assessment
- Generating a learning pathway
- Identifying Champion candidates
- Creating a peer-led session
- Logging behaviour change evidence
- Fetching AMP impact dashboard data

10. Recommended Tech Stack
Recommend a hackathon-friendly stack.
Prioritise speed, reliability, and demo quality.
Consider:
- Frontend
- Backend
- Database
- Authentication
- AI/LLM integration
- Analytics/dashboarding
- Deployment

Explain why each choice is appropriate.

11. Repository Structure
Propose a clean repo layout.
Include folders for:
- Frontend app
- Backend/API
- Shared types
- AI prompts/services
- Seed data
- Programme content
- Tests
- Documentation

12. Implementation Plan
Break the build into hackathon milestones:
- First 2 hours
- First half-day
- Day 1
- Final demo preparation

Include task assignments for a small team of 3-5 people.

13. Demo Reliability Strategy
Explain how to make the project robust for judging.
Include:
- Seed data
- Mock AI fallback
- Preloaded assessment examples
- Offline/local fallback
- Dashboard screenshots or cached results
- Simple failure handling

14. Risks and Tradeoffs
Identify key risks, such as:
- Over-reliance on AI-generated recommendations
- Measuring attendance instead of behaviour change
- Privacy concerns
- Cultural tokenism
- Hard-to-scale peer programmes
- Weak Champion selection criteria

Suggest practical mitigations.

15. Final Output
End with:
- A concise architecture summary
- The recommended MVP scope
- The strongest demo narrative
- What should be built versus mocked during the hackathon

Be specific, structured, and pragmatic. Avoid generic AI education advice. Focus on a solution AMP could realistically pilot with Muslim professionals and scale through workplaces and community spaces.
